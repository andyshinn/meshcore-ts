# Documentation Site: TypeDoc + Starlight — Design

**Date:** 2026-06-15
**Status:** Approved (design); pending spec review
**Topic:** Documentation website for `@andyshinn/meshcore-ts`

## Goal

Stand up a documentation website for the `@andyshinn/meshcore-ts` library that:

1. **Describes the API** — an auto-generated reference covering the full public
   surface exported from `src/index.ts` (session, ~20 feature modules, ports,
   codes, errors, types).
2. **Provides basic examples** — a small set of hand-written guides (getting
   started, implementing a transport, messaging, events & state) ported and
   expanded from the existing `README.md`.

The site combines **TypeDoc** (API extraction) with **Astro Starlight** (the
docs theme/site), unified into a single navigable site rather than two separate
UIs.

## Decisions (locked during brainstorming)

| Decision | Choice |
| --- | --- |
| Site location | `docs/` as the Astro/Starlight project root |
| Deployment | GitHub Pages via a GitHub Actions workflow (dormant until pushed to GitHub + Pages enabled) |
| API scope | Full public surface auto-generated from `src/index.ts` |
| Integration mechanism | `starlight-typedoc` plugin (runs TypeDoc + `typedoc-plugin-markdown` inside the Astro build) |

## Approach

Use the **`starlight-typedoc`** plugin (by HiDeoo). It runs TypeDoc with
`typedoc-plugin-markdown` as part of the Astro build, emitting Markdown directly
into Starlight's content collection and exposing a `typeDocSidebarGroup` object
to slot into the Starlight sidebar. This yields one site, one navigation, one
theme — "TypeDoc combined with Starlight."

Alternatives rejected:

- **Standalone TypeDoc HTML linked from Starlight** — two separate UIs; not
  "combined."
- **Manual `typedoc-plugin-markdown` prebuild script** — reinvents what
  `starlight-typedoc` already wraps; more moving parts.

## Architecture

The docs site is a **self-contained npm project** rooted at `docs/`, with its
own `package.json`, lockfile, and `node_modules`. It is intentionally **not**
linked to the root library package (no npm workspaces). Rationale:

- **Isolation of the docs toolchain.** Astro, Starlight, TypeDoc, and their
  transitive deps stay out of the library's dependency tree.
- **Contains the TypeScript-version risk.** TypeDoc supports "the two latest
  TypeScript releases." The library pins `typescript@6.0.3` (very new). The
  `docs/` project carries its own pinned `typescript` at whatever version the
  chosen TypeDoc release supports, so a TypeDoc↔TS peer-range mismatch cannot
  break the library build. TypeDoc reads the library source via
  `../src/index.ts` and `../tsconfig.json` but uses the docs project's own
  TypeScript to do so.

### Directory layout

```
docs/
  package.json              # docs-only deps + scripts
  astro.config.mjs          # Starlight + starlightTypeDoc plugin config
  tsconfig.json             # extends astro/tsconfigs/strict
  src/
    content.config.ts       # Starlight content collection (docsLoader/docsSchema)
    content/
      docs/
        index.mdx           # landing / splash page
        guides/
          getting-started.md
          transports.md     # implementing a Transport (BLE/serial sketch)
          messaging.md      # DM + channel send, message-state lifecycle
          events-and-state.md
        api/                # GENERATED by starlight-typedoc (gitignored)
  superpowers/              # existing specs/plans — untouched; outside Astro's content dir
```

Astro's content lives under `docs/src/content/docs/`, so the existing
`docs/superpowers/` directory is a sibling Astro ignores.

### Configuration

`docs/astro.config.mjs` (shape; exact options finalized in the plan):

```js
import { defineConfig } from 'astro/config'
import starlight from '@astrojs/starlight'
import starlightTypeDoc, { typeDocSidebarGroup } from 'starlight-typedoc'

export default defineConfig({
  site: 'https://andyshinn.github.io',
  base: '/meshcore-ts',
  integrations: [
    starlight({
      title: 'meshcore-ts',
      social: [{ icon: 'github', label: 'GitHub', href: 'https://github.com/andyshinn/meshcore-ts' }],
      plugins: [
        starlightTypeDoc({
          entryPoints: ['../src/index.ts'],
          tsconfig: '../tsconfig.json',
          // output defaults to 'api' (relative to content/docs)
        }),
      ],
      sidebar: [
        {
          label: 'Guides',
          items: [
            { label: 'Getting started', slug: 'guides/getting-started' },
            { label: 'Transports', slug: 'guides/transports' },
            { label: 'Messaging', slug: 'guides/messaging' },
            { label: 'Events & state', slug: 'guides/events-and-state' },
          ],
        },
        typeDocSidebarGroup, // generated API reference group
      ],
    }),
  ],
})
```

`entryPoints` and `tsconfig` are resolved relative to the Astro project root
(`docs/`), hence `../`.

The `site`/`base` values assume the GitHub repo path `andyshinn/meshcore-ts`
(Pages URL `https://andyshinn.github.io/meshcore-ts/`). These are trivially
editable if the actual repo path differs.

### Content plan

Hand-written guides (ported/expanded from `README.md`):

- **index.mdx** — splash landing page: one-line pitch, install command, links to
  Getting started and the API reference.
- **guides/getting-started.md** — install, construct a `MeshCoreSession` with a
  `LoopbackTransport`, subscribe to events before connecting, run the handshake,
  send a first message. (Mirrors the README "Quick start".)
- **guides/transports.md** — the `Transport` port contract (one complete
  companion frame per `onData`), the `LoopbackTransport` for tests, and the
  real-`BleTransport` sketch.
- **guides/messaging.md** — `sendDmText` / `sendChannelText`, the
  `sending → sent → ack` message-state lifecycle via the `messageState` event,
  and `registerChannelSend` for path attribution.
- **guides/events-and-state.md** — the typed event emitter, the full event list,
  and reading `session.state`.

Generated API reference (auto): every export from `src/index.ts`, written to
`docs/src/content/docs/api/` and surfaced via `typeDocSidebarGroup`.

### Deployment

`.github/workflows/docs.yml`:

- Trigger: `push` to `main` (plus `workflow_dispatch`).
- Steps: checkout → setup Node 20 → `npm ci` in `docs/` → `astro build`
  (which regenerates the API reference) → upload artifact →
  `actions/deploy-pages`.
- Permissions: `pages: write`, `id-token: write`; `concurrency` group for Pages.

Dormant until the repo is pushed to GitHub and Pages is enabled
(Settings → Pages → Source: GitHub Actions).

### Developer experience

Root `package.json` gains pass-through scripts (no `cd`, uses `--prefix`):

```json
{
  "scripts": {
    "docs:dev": "npm --prefix docs run dev",
    "docs:build": "npm --prefix docs run build",
    "docs:preview": "npm --prefix docs run preview"
  }
}
```

`docs/package.json` scripts: `dev` (`astro dev`), `build` (`astro build`),
`preview` (`astro preview`).

### Ignored / generated files

Add to root `.gitignore`:

```
docs/.astro/
docs/dist/
docs/src/content/docs/api/
```

(`node_modules/` and `dist/` patterns already match nested paths; `docs/dist/`
is listed explicitly for clarity.) The generated API Markdown is **not**
committed — it is rebuilt on every `astro build`, including in CI.

## Components & boundaries

| Unit | Responsibility | Depends on |
| --- | --- | --- |
| `docs/astro.config.mjs` | Site config, Starlight theme, plugin wiring, sidebar | Starlight, `starlight-typedoc` |
| `starlight-typedoc` plugin | Run TypeDoc → Markdown into content collection | TypeDoc, `typedoc-plugin-markdown`, `../src/index.ts`, `../tsconfig.json` |
| Hand-written guides | Narrative docs + examples | none (static MDX/MD) |
| `.github/workflows/docs.yml` | Build + deploy to GitHub Pages | `docs/` project, Pages |
| Root pass-through scripts | Run docs tasks from repo root | `docs/` project |

## Error handling / risks

- **TypeDoc ↔ TypeScript 6.0 peer range.** Primary risk. Mitigation: isolated
  `docs/` dependency tree with its own pinned TypeScript; verified empirically
  by running `astro build` (which invokes TypeDoc) during implementation. If the
  current TypeDoc release does not yet accept TS 6.0, pin `docs/`'s `typescript`
  to the latest version TypeDoc supports — the library source is unlikely to use
  TS 6.0-only syntax that an older parser rejects, but this is checked at build.
- **`@types/node` resolution for TypeDoc.** The library `tsconfig.json` sets
  `"types": ["node"]`; the docs project includes `@types/node` (or relies on
  upward resolution) so TypeDoc can resolve node type references.
- **Pages base path mismatch.** If the repo path differs from
  `andyshinn/meshcore-ts`, update `site`/`base` in `astro.config.mjs`.

## Testing / verification

- `npm --prefix docs install` then `npm run docs:build` completes without error
  and produces `docs/dist/` plus a populated `docs/src/content/docs/api/`.
- `npm run docs:dev` serves the site locally; Guides and the API reference both
  render and are navigable.
- `npm run lint` / `npm run typecheck` / `npm test` at the repo root remain
  green (docs setup must not touch the library build).

## Out of scope (YAGNI)

- Custom Starlight theme/branding beyond title + GitHub link.
- Versioned docs / multiple released versions.
- Search backends beyond Starlight's built-in Pagefind.
- Hosting on Vercel/Netlify/Cloudflare (GitHub Pages chosen).
- Converting the repo to npm workspaces.
